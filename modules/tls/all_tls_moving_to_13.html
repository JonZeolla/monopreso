Moving to TLS 1.3

Issues with prior TLS versions

  1. Easy to monitor the activities of clients.
  2. Often able to downgrade clients security to use breakable encryption, called export ciphers.
  3. Supports features which have been the source of attacks on TLS
    MITM attacks on TLS 1.2
  1. Cipher suites areâ€™t digitally signed, and could be downgraded by a MITM. There is an interim mitigation called SCSV (RFC 7507) which has gotten partial adoption. Because TLS 1.2 (and prior version) supports weak encryption and hashing, this could be a significant issue.
    SCSV
  1. For TLS 1.2, SCSV adds a downgrade canary into the client hello message if it's retrying due to a network error (potential MITM/injection). For TLS 1.3 this is not needed, as downgrade protection is built into the protocol
    Features removed in TLS 1.3 due to security concerns
  1. Features, such as compression have been the cause of issues in the past.
    The deprecation of TLS 1.0 and 1.1 is in progress
  1. draft-ietf-tls-oldversions-deprecate-05
  2. Firefox has published their intent to deprecate on 9/12/19.

Issues with prior TLS versions

  1. Compromise of a secret key may result in the ability to decrypt prior communications

Benefits of TLS 1.3

  1. Client certificates may be encrypted to prevent client tracking
  2. The cipher suites offered by the server are now digitally signed
  3. Numerous TLS 1.3 implementations have been formally verified to be secure
  4. TLS 1.3 has removed much backwards compatability in favor of removing insecure or error-prone features
  • Client certificate tracking is a concern with large scale monitoring.
  • The TLS 1.3 formal verifications were done with Tamarin, Proscript-TLS, miTLS, and nqsb-TLS. More details were presented here.
